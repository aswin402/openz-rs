# OpenZ Security Guard & Permissions Guide 🔒🛡️

OpenZ implements a robust, pluggable **Security Guard Interceptor** that acts as an active gatekeeper for the host system. It intercepts any sensitive tool calls generated by the AI agent and blocks execution until the user explicitly grants permission.

---

## 🚫 Intercepted Tool Categories

The guard matches commands using whole-token word detection to prevent bypasses and false positives. The following categories of actions will trigger an approval prompt:

### 1. Destructive Commands
* **Filesystem Deletion**: `rm`, `rmdir`, `unlink`.
* **Low-level Write/Erase**: `dd`.
* **Filesystem Manipulation**: `mkfs`, `fdisk`, `parted`, `format`.
* **Build Clean Operations**: `cargo clean`, `npm run clean`, `bun run clean`, `yarn clean`.

### 2. Privilege Escalation
* **Superuser Execution**: `sudo`, `su`.
* **Permission Changes**: `chmod`, `chown`.

### 3. Process Control
* **Process Termination**: `kill`, `killall`, `pkill`.

### 4. System Control
* **Host Power State**: `shutdown`, `reboot`, `poweroff`, `halt`.

### 5. Network Transfers (Download/Upload)
* **Remote File Fetching & Posting**: `curl`, `wget`.
* **Remote Copies**: `scp`, `rsync`, `sftp`.
* **Raw Sockets**: `nc`, `netcat`.

### 6. Out-of-Workspace Writing
* Any native `write_file` tool call targeted outside the configured `/workspace` directory or the user's active `/home/` directory (e.g. attempting to edit files in `/etc/`, `/usr/bin/`, or `/var/`).

## 💻 Console CLI / TUI Approval Menu

When a sensitive command is requested in the terminal, OpenZ displays a clean, minimal themed select menu that matches the style of the `/model` selector:

```text
🔒 SECURITY SHIELD: Sensitive Action Requested
  Tool:      exec_command
  Details:   $ rm -rf /workspace/tmp/cache
> Authorize execution?
  Approve (Allow once)
  Approve & Trust for this session
  Deny (Abort tool)

  ↑/↓ Navigate · enter Select
  esc to cancel                                  Security Shield
```

### Options:
1. **`Approve (Allow once)`**: Authorizes only this single command execution. Future calls will prompt again.
2. **`Approve & Trust for this session`**: Authorizes this command and whitelist/trusts this tool (e.g., `exec_command` or `write_file`) for the rest of the current active session. All subsequent sensitive calls for this tool will run automatically without prompting.
3. **`Deny (Abort tool)`**: Aborts the execution. The agent loop yields a standard error response: `{"error": "Execution denied by user."}` so the model knows the action was rejected.

## 🤖 Telegram Inline Callback Approvals

If OpenZ is running as a background channel (e.g. over Telegram), the approval request is sent as an interactive message containing inline keyboard buttons:

1. OpenZ sends a formatted markdown message:
   > ⚠️ **SECURITY ALERT**
   > OpenZ is requesting to execute a sensitive tool `exec_command`:
   > ```
   > $ rm -rf /workspace/tmp/cache
   > ```
   > Do you approve this action?
   > `[ Approve ✅ ]`   `[ Deny ❌ ]`
2. The agent loop suspends execution asynchronously using a `tokio::sync::oneshot` channel waiting thread.
3. When the user clicks an inline button, the Telegram polling listener parses the `callback_query` update, resolves the oneshot channel, answers the query to stop client loading, and updates the inline markup to display a disabled **`Approved ✅`** or **`Denied ❌`** state.
